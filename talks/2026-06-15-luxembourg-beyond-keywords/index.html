One historian, no team — manual cataloguing is impossible. AI already builds it (OCR, entity recognition). Today is about using AI to access it.

For fifteen years I've worked on Islam in Francophone West Africa — mostly Benin, Burkina Faso, Côte d'Ivoire, Togo. Lots of interviews, but also extensive work with West African newspapers and Islamic publications. I'm a digital hoarder: 30,000+ items in Zotero, ~15,000 unread. I wanted them in an open-access database, reached agreements with copyright holders, and launched the IWAC in 2023. It now holds 14,700+ documents on Muslim public life across six countries, ~96% in French. I built and maintain it alone — which is exactly why AI matters here: LLMs already do the work a team would (OCR on degraded scans, entity recognition, sentiment). So far I've used AI to build the collection; today is about opening it.

The "El Hadj" problem

Why keyword search fails

Search "hadj" and you get two unrelated things at once:

  • the pilgrimage to Mecca
  • the hundreds of men who carry "El Hadj" as an honorific
  • One imam — 20+ spellings across newspapers and decades
  • Search "hajj" and you'll never find pèlerinage à La Mecque
  • The engine works exactly as designed; the multilingual lexicon defeats it
The gap I've known this corpus since 2015 — its tags, its quirks. The search box assumes you do too.
I want the collection accessible to researchers and the general public alike. But in the IWAC, "hadj" returns two completely different things: articles about the pilgrimage, and the hundreds of men who carry "El Hadj" as an honorific after making it. A single imam can appear under 20+ spellings. The search engine is doing exactly what it was built to do — what defeats it is the vocabulary: synonymy, polysemy and transliteration all at once, on top of degraded OCR. Keyword boxes assume you type one query and judge a ranked list produced by an algorithm you can't see. I've known this corpus since 2015 — its organisation, its tags. The users don't.

Model Context Protocol · open standard, Nov 2024

Why a server, not just a chatbot?

RAG

Copies the collection into an external store — chunked, embedded, retrieved by a similarity the user never sees.

MCP

Ingests nothing. The data stays put; the server holds the logic; the model just asks and answers.

The analogy Like IIIF for images: a shared interface, without handing over the originals. Build the server once — Claude, ChatGPT, or open-source models all connect.
MCP is an open standard, released November 2024, for connecting AI assistants to external data through a defined set of tools. A server publishes the queries it will answer; any compliant assistant can call them. MCP is to collection data what IIIF is to images — IIIF succeeded by solving a coordination problem: institutions exposed high-resolution images through a shared interface without handing over the originals. Why not RAG? RAG copies the collection into a vector store, chunks it, retrieves by similarity the user never sees. Fine for many uses — but for an institution that cares about data sovereignty (and African collections have a long history of extractive knowledge practices), moving the data into someone else's pipeline isn't neutral. MCP draws the line differently: nothing is ingested, chunked, or used for training. Build once, and any model — commercial or open-source — can be the interface.

A skill is a methods handbook, in plain language, that the agent reads into context when a request matches its purpose. The model gains a specific competence by reading, not retraining. The difference from a prompt is durability and accountability: a prompt is typed once and lost; a skill file is versioned, reviewed, published, refined — tacit expertise turned into an explicit, inspectable artefact. Like MCP, skills are an open standard across agents, so nothing locks you to one vendor. They're mostly used for coding, but work just as well for the social sciences: how to behave like a historian, what to know about a corpus.

read-only verifiable one-click install

~20 tools
  • Search · full text · authority records · sentiment · stats
  • Answers from a local copy of the open IWAC dataset
  • No AI at query time — every call is a structured lookup
  • Every result links to the canonical record
What I actually built: some twenty tools — article search by keyword, country, newspaper, subject and date; full-text retrieval; authority records for persons, places, organisations and events; precomputed sentiment; collection statistics. The design principle is verifiability. The server does no generation at query time: every AI enrichment was computed offline and stored as ordinary data, so a tool call is only ever a lookup against the openly published dataset — and every result carries a URL straight back to the canonical IWAC record. In Sternfeld's phrase, a walled garden: the AI can cite normally only what the collection actually contains. One-click install for Claude Desktop, nothing to administer — which matters if the point is access for non-specialists. Now let me show you.

The IWAC skill in motion

A five-phase research method

  1. Scoping — what does the collection even hold for this question, before any keyword?
  2. Systematic searching — French + transliteration variants (Tabaski = Eid al-Adha); log every search, including null results
  3. Deep reading — full text and sentiment, article by article
  1. Triangulation — cross-check articles, publications, references, index entries
  2. Synthesis — findings with source attribution and confidence grades
The point Twenty tools give reach, not judgement. The judgement lives in the skill.
Beyond the server, I wanted the model to know how to behave — to transmit my own knowledge of the collection as a historian working with primary sources. The server alone knows none of it: not that every search must be in French, not that "Côte d'Ivoire" without its circumflex returns zero, not that Niger's coverage is thin and there's almost no Nigerian press. Twenty tools give reach, not judgement. That knowledge lives in the skill: a five-phase workflow drawn from how archival research actually works. Scoping first — what's even here. Then systematic searching in French with transliteration variants built in (Tabaski and Aïd el-Kébir both mean Eid al-Adha), every search logged including the null ones. Deep reading. Triangulation across source types. Synthesis with explicit sources and confidence grades — strong, moderate, weak.
